
#  Pixel Arithmetic and Data Types

## 📌 مقدمه

یکی از رایج‌ترین عملیات‌ها در پردازش تصویر، تغییر مقدار پیکسل‌ها است. برای مثال:

- افزایش روشنایی (Brightness)
- کاهش روشنایی
- افزایش کنتراست
- اعمال فیلترها
- انجام بسیاری از الگوریتم‌های پردازش تصویر

اما قبل از انجام این محاسبات باید بدانیم که **مقادیر پیکسل‌ها چگونه در حافظه ذخیره می‌شوند**؛ زیرا نوع داده (Data Type) تعیین می‌کند هر پیکسل چه مقادیری را می‌تواند نگه دارد.

---

# 🔹 نوع داده uint8 چیست؟

در OpenCV تقریباً تمام تصاویر معمولی با نوع داده **uint8** ذخیره می‌شوند.

کلمه **uint8** از سه بخش تشکیل شده است:

- **u** → Unsigned (بدون علامت)
- **int** → Integer (عدد صحیح)
- **8** → 8 بیت

یعنی هر پیکسل یک عدد صحیح 8 بیتی بدون علامت است.

---

## 📌 محدوده مقادیر در uint8

از آنجایی که برای هر پیکسل فقط 8 بیت در نظر گرفته شده است، تعداد حالت‌های ممکن برابر است با:

$$
2^8 = 256
$$

بنابراین هر پیکسل تنها می‌تواند یکی از 256 مقدار زیر را داشته باشد:

```
0 , 1 , 2 , ... , 254 , 255
```

در نتیجه:

- کوچک‌ترین مقدار برابر **0** است.
- بزرگ‌ترین مقدار برابر **255** است.
- هیچ عددی کمتر از **0** وجود ندارد.
- هیچ عددی بزرگ‌تر از **255** نیز قابل ذخیره نیست.

برای تصاویر خاکستری:

- **0** → کاملاً سیاه
- **255** → کاملاً سفید
- اعداد بین آن‌ها → درجات مختلف خاکستری

---

# ☀️ اگر بخواهیم تصویر را روشن‌تر کنیم چه اتفاقی می‌افتد؟

فرض کنید مقدار یک پیکسل برابر باشد با:

```
120
```

اگر بخواهیم روشنایی تصویر را 30 واحد افزایش دهیم، کافی است به مقدار همه پیکسل‌ها عدد 30 را اضافه کنیم.

```
120 + 30 = 150
```

این مقدار هنوز داخل بازه 0 تا 255 قرار دارد، بنابراین مشکلی وجود ندارد.

---

اما حالا فرض کنید مقدار یک پیکسل برابر باشد با:

```
240
```

اگر دوباره 30 واحد به آن اضافه کنیم:

```
240 + 30 = 270
```

در حالی که **270 در uint8 وجود ندارد.**

زیرا بزرگ‌ترین عدد قابل ذخیره **255** است.

پس سؤال مهم اینجاست:

**کامپیوتر با این مقدار چه کار می‌کند؟**

---

# ⚠️ مشکل Overflow

وقتی نتیجه یک محاسبه از بیشترین مقدار قابل ذخیره بیشتر شود، به این حالت **Overflow** گفته می‌شود.

در نوع داده uint8، مقدار 270 قابل ذخیره نیست.

بنابراین اگر مستقیماً روی تصویر عملیات انجام دهیم، نتیجه ممکن است اشتباه شود و مقدار پیکسل تغییر غیرمنتظره‌ای داشته باشد.

به همین دلیل نباید محاسبات را مستقیماً روی uint8 انجام دهیم.

---

# 🌑 اگر بخواهیم تصویر را تیره‌تر کنیم چه می‌شود؟

فرض کنید مقدار یک پیکسل برابر باشد با:

```
10
```

اگر بخواهیم 20 واحد از آن کم کنیم:

```
10 - 20 = -10
```

اما در uint8 عدد منفی وجود ندارد.

در نتیجه دوباره با یک مقدار غیرمجاز روبه‌رو می‌شویم.

این حالت را **Underflow** می‌نامند.

---
# ⚠️ مشکل Overflow و Underflow (رفتار واقعی uint8)

## 🔺 Overflow در uint8

وقتی مقدار از 255 بیشتر شود، اتفاقی که خیلی مهم است رخ می‌دهد:

در uint8 عددها **نمی‌مانند و دوباره از صفر شروع می‌شوند (Wrap Around)**.

یعنی سیستم عدد را به جای اینکه نگه دارد، آن را به صورت چرخشی برمی‌گرداند.

---

### 📌 مثال:

```
240 + 30 = 270
```

اما در uint8:

```
270 → 270 - 256 = 14
```

یعنی مقدار نهایی می‌شود:

```
14 ❌ (نه 255 یا 270)
```

---

## 🌑 Underflow در uint8

برای مقادیر منفی هم همین اتفاق می‌افتد.

مثلاً:

```
10 - 20 = -10
```

اما در uint8 عدد منفی نداریم، پس سیستم دوباره به صورت چرخشی عمل می‌کند:

```
-10 + 256 = 246
```

یعنی نتیجه می‌شود:

```
246 ❌ (کاملاً اشتباه نسبت به انتظار ما)
```

---

## 🚨 چرا این موضوع خیلی مهم است؟

این رفتار باعث یک مشکل جدی در تصویر می‌شود:

- رنگ‌ها به‌جای روشن‌تر شدن، ناگهانی تغییر می‌کنند
- تصویر دچار **distortion** می‌شود
- رنگ‌ها غیرواقعی و غیرطبیعی دیده می‌شوند
- ساختار روشنایی تصویر به هم می‌ریزد

📌 یعنی به جای اینکه تصویر فقط روشن‌تر یا تیره‌تر شود، کل توزیع رنگ‌ها **خراب می‌شود**

---

### 🎯 نتیجه مهم

به همین دلیل:

❌ نباید هیچ عملیات ریاضی را مستقیم روی `uint8` انجام دهیم  
چون نتیجه واقعی محاسبه از دست می‌رود و تصویر خراب می‌شود

---

# 🧠 راه‌حل چیست؟

برای جلوگیری از این مشکل باید:

1. تصویر را از `uint8` خارج کنیم
2. به یک نوع داده بزرگ‌تر و امن‌تر ببریم (مثل `int16`)
3. محاسبات را انجام دهیم
4. سپس مقدارها را به محدوده 0 تا 255 برگردانیم

---



# توضیح راه‌حل


قبل از انجام محاسبات، نوع داده تصویر را به **int16** تبدیل می‌کنیم.

---

در int16:

- اعداد منفی وجود دارند.
- اعداد بسیار بزرگ‌تر از 255 نیز قابل ذخیره هستند.

محدوده آن تقریباً برابر است با:

```
-32768  تا  32767
```

حالا اگر بنویسیم:

```
240 + 30
```

نتیجه برابر خواهد بود با:

```
270
```

و بدون هیچ مشکلی ذخیره می‌شود.

همچنین:

```
10 - 20
```

برابر می‌شود با:

```
-10
```

که آن هم کاملاً قابل ذخیره است.

بنابراین تمام محاسبات ابتدا در محیطی انجام می‌شوند که محدودیت uint8 را ندارد.

---


اما تصویر نهایی باید دوباره از نوع uint8 باشد.

پس لازم است تمام مقادیر را به بازه مجاز برگردانیم.

این کار با تابع **clip** انجام می‌شود.

قانون آن بسیار ساده است:

- اگر مقدار کمتر از 0 باشد → برابر 0 می‌شود.
- اگر مقدار بیشتر از 255 باشد → برابر 255 می‌شود.
- اگر داخل بازه باشد → بدون تغییر باقی می‌ماند.

برای مثال:

```
-15  → 0

120  → 120

280  → 255
```

---

# 🔄 چرا دوباره به uint8 برمی‌گردیم؟

اکثر توابع OpenCV، نمایش تصویر با Matplotlib و همچنین ذخیره تصاویر با فرمت‌هایی مانند JPEG و PNG انتظار دارند تصویر از نوع uint8 باشد.

بنابراین پس از پایان محاسبات:

1. تصویر را به int16 تبدیل می‌کنیم.
2. عملیات موردنظر را انجام می‌دهیم.
3. مقادیر را با clip محدود می‌کنیم.
4. دوباره تصویر را به uint8 تبدیل می‌کنیم.

این دقیقاً همان روشی است که در بسیاری از پروژه‌های واقعی پردازش تصویر و بینایی ماشین استفاده می‌شود.

---

# 🚀 جمع‌بندی

برای جلوگیری از بروز خطا در محاسبات پیکسلی، هیچ‌گاه عملیات افزایش یا کاهش شدت نور را مستقیماً روی تصویر uint8 انجام نمی‌دهیم.

روند استاندارد به صورت زیر است:

```
Image (uint8)
       │
       ▼
Convert to int16
       │
       ▼
Pixel Arithmetic (+ , - , × , ...)
       │
       ▼
Clip values to [0,255]
       │
       ▼
Convert back to uint8
```

این روش از بروز خطاهای **Overflow** و **Underflow** جلوگیری کرده و باعث می‌شود نتیجه محاسبات کاملاً صحیح باشد.
